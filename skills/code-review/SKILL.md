# code-review

## 代码审查步骤

### 第一步：了解用户变更

先获取暂存区文件列表：

```bash
git diff --staged --name-status
```

- 如果暂存区为空，输出「暂存区无变更，审查终止」并停止
- **不要执行任何修改暂存区的命令**，如 `git add`
- 暂存区外的文件不属于审查对象

再逐文件查看具体改动：

```bash
git diff --staged -- <file>
```

### 第二步：获取相关代码用于辅助审查

在暂存区变更的基础上，读取相关代码以辅助判断

- 对于修改的函数/方法，读取其完整定义
- 对于接口、类、结构体等定义变更，读取整个定义文件
- 存在跨文件调用时读取调用方或被调用方的相关代码，最多追溯 2 层
- 相关代码仅用于理解变更，不在审查范围内

### 第三步：审查并输出报告

参照 [references/check-items.md](references/check-items.md) 中的检查项，逐项审查

- 仅报告至少能举出具体反例或引用具体规范条款的问题
- 不确定的问题标注为存疑，并提供存疑原因

审查完成后输出报告，格式如下：

| 序号 | 文件         | 行号  | 问题         | 修复建议                      |
| ---- | ------------ | ----- | ------------ | ----------------------------- |
| 1    | query.ts     | 42-44 | SQL 注入     | 使用参数化查询                |
| 2    | reader.py    | 78    | 资源未关闭   | 使用上下文管理器或 defer 关闭 |
| 3    | Service.java | 103   | 遗留调试日志 | 移除或改用日志库              |

- 如果无问题，输出「本次审查未发现问题」
- 不得添加前言、总结性文字等额外内容
